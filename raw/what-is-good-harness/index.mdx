## 모델(말)과 하네스(마구)


## LLM이 할 수 없는 것

**LLM은 메모리를 갖지 않는다.** 
입력 -> 추론 -> 출력을 하는 동작만 수행할 뿐이다. 
따라서 
- 이 프로젝트의 Guardrail
- Plan
- Convention
같은 것들은 스스로 알아내지 못한다.

**LLM은 행동할 수 없다.**
마찬가지로 출력할 것을 예측해 반환할 뿐이다.
따라서
- 파일 읽기
- 검색하기
- 테스트 실행하기
같은 행동은 수행할 수 없다.

이러한 특성을 이해한다면 우리는 하네스가 LLM의 어떤 부분을 채워줘야하는지 이해할 수 있다.
## 하네스의 역할

**Memory(기억장치)**

LLM은 기억하는 능력이 없으므로 하네스가 LLM의 기억장치가 되어줘야한다.
- 대화 이력
- AGENTS.md

**Capability(실행능력)**

LLM은 행동할 수 없으므로 하네스가 LLM의 손,발이 되어줘야한다.
- Tool(Search, Read, Test 등)

## 좋은 하네스의 조건

그렇다면 좋은 하네스는 '좋은 메모리와 좋은 실행능력을 갖추었다.'로 생각할 수 있다.
다시 좋은 기억장치와 좋은 실행능력은 무엇인지 정의해보자.

### 좋은 메모리
좋은 메모리란 무엇인가 따져보기 위해 우리가 LLM으로 무엇을 하고자 하는지 생각해보자. 우리가 LLM을 사용해 얻고자 하는 것은 '좋은(올바른, 만족스러운) 결과물'이다.

우리는 이미 chatGPT, Gemini등을 사용해오며 프롬프트 엔지니어링이 중요하다는 것을 이미 알고 있다. 즉, 개떡같이 물어보면 개떡같이 답하고 찰떡같이 물어보면 찰떡같이 답한다는 것이다.

> 어떤 메모리가 나쁜 결과를 만드는가?

LLM이 나쁜 결과물을 만드는 경우를 다음과 같이 정리해볼 수 있다.

**LLM은 학습되지 않은 정보를 추론할 수 없다.**

불행히도 LLM은 모른다고 말하기보다 어떻게든 아는척을 할 때 칭찬을 받도록 학습되었다. (불행히도 인간도 그걸 더 좋아했다.) 때문에 LLM은 모르는 것을 물어볼 때 '몰라'라고 답하지 않고 확률 낮은 추측을 한다.

```
Q: 우리집 현관문 비밀번호는?
A: 1234 아닐까요?
```


비슷하게 코딩 에이전트 관점에서도 "현재 프로젝트의 인증 과정은 어떻게 돼?"라는 질문에 아무 컨텍스트도 주지 않는다면 "auth.ts에 있을 것 같아요(50~60%정도?)"라는 추측을 하지만 "login.ts" 파일을 컨텍스트에 넣어준다면 "login.ts에 있어요(99%로)" 라고 답할 것이다.

**LLM은 때때로 잊는다.**

```
Q: 앞으로 모든 대답은 한국어로 대답해.
A: 네, 앞으로 모든 대답은 한국어로 할게요.

...(대충 긴 컨텍스트)

Q: 이 문서를 요약해봐.
A: Sure, here's a summary...
```

실제로 요즘 Claude가 계속 일본어를 해서 조금 곤란하다. (물론 이 문제가 아닐 수 있다.)
LLM은 모든 입력을 밀도있게 다룰 수 없기 때문에 종종 오래되고 중요도가 낮다고 판단해버리면 추론 과정에 해당 정보를 사용하지 않기도 한다. 

**LLM은 우선순위를 스스로 결정하지 못한다.**

```
안정성을 우선할 것

...

가능한 빠르게 구현해봐
```

사람한테 이렇게 시켜도 곤란하다. LLM은 충돌하는 조건이 주어질 때 무엇을 우선해야하는지 판단할 수 없다. 물론 사람의 일도 때때로 충돌되는 조건을 마주하기도 한다. 이때 무엇을 우선할지 기준이 없다면 끔찍한 회의의 늪에 끌려갈지도 모른다.

이제 나쁜 메모리의 조건을 알았으니 좋은 메모리의 조건도 정리가 가능할 것 같다.
1. 필요한 정보를 포함한다.
2. 중요한 정보는 강조된다.
3. 우선순위를 포함한다.

### 좋은 실행능력

이 글에서 필자는 엄밀한 검증보다는 납득할 수 있는 논리를 피고싶기 때문에 어느정도 당연히 그러하다는 가정을 깔고 가려고 한다.

**당연한 가정: 볼 수 있고, 쓸 수 있다**

인간에게 눈,귀와 같은 감각기관과 손,발 같은 운동기관이 없었다면 좀 슬펐을 것이다. 감각기관이 있어야 필요한 정보를 얻을 수 있고, 운동기관이 있어야 행동할 수 있다. 이것은 실행능력의 필요조건으로 보며 왜 필요한지 까지는 구태여 따지진 않겠다. 하지만 그것으로 충분한가? 눈이 100개 있고, 손이 100개 있다면 그것은 좋은 실행 능력을 갖췄다 할 수 있는가?

**검증할 수 있다**

아무리 많은 행동을 수행할 수 있어도 그 결과를 검증할 수 없다면 실행은 우연에 의존하게 된다. 원숭이가 나무에서 떨어지듯 LLM은 할루시네이션을 일으키고, 이것은 LLM 스스로도 제어할 수 없다. 따라서 우리는 올바른 실행을 했는지 검증해야한다. 이는 TDD에서 말하는 "좋은 소프트웨어는 테스트 가능하다"는 관점과도 일치한다.

다시 말해 좋은 실행능력의 핵심은 "검증 루프"이다. LLM이 실수하지 않게 만드는 것이 아니라, 실수를 발견하고 수정할 수 있게 만드는 마지막 안전장치라 할 수 있다.
