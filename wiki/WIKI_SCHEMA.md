# Wiki Schema

이 wiki는 블로그 포스트들로부터 **"개발자로서 나는 어떤 사람인지"**를 구조화하는 지식 레이어다.

블로그의 목표는 "나의 성장을 기록하는 것"이다. 포스트에는 개인적인 생각, 리뷰, 회고, 팩트 기반 지식이 섞여있다. wiki는 이것들을 읽고 정리하여 다음 질문에 답할 수 있게 한다:

- 나는 어떤 사람인지, 어떻게 생각하는 사람인지
- 어떻게 성장하고 있고, 생각이 어떻게 바뀌어왔는지
- 앞으로 무엇을 탐구하면 좋을지

## 원칙

- **raw/는 읽기 전용**: wiki는 `raw/`의 포스트를 읽기만 한다. 절대 수정하지 않는다.
- **블로그 빌드와 무관**: wiki는 블로그 렌더링에 관여하지 않는다. wiki를 삭제해도 블로그는 정상 동작한다.
- **파생 산출물**: `raw/`가 source of truth이고, wiki는 그로부터 파생된 지식 레이어다.
- **공개된 포스트만 대상**: `draft: false`인 포스트만 wiki에 포함한다. draft 상태의 포스트는 ingest하지 않는다.
- **표준 markdown**: 특정 도구에 종속되지 않는 표준 markdown 링크(`[텍스트](경로)`)를 사용한다.

## 디렉토리 구조

```
wiki/
├── WIKI_SCHEMA.md    # 이 파일
├── index.md          # 전체 wiki 페이지 카탈로그
├── log.md            # 작업 이력 (시간순, append-only)
├── sources/          # 포스트별 요약 + 관계 + 맥락
└── topics/           # 주제별 군집 + 종합 인사이트
```

## Source 페이지 (`sources/{folder-name}.md`)

포스트 하나에 대응하는 문서. 단순 요약이 아니라, 그 포스트에서 드러나는 생각과 다른 포스트들과의 접점을 함께 담는다.

파일명은 포스트 폴더명과 동일하다. 원본은 `raw/{folder-name}/index.mdx`.

### frontmatter

```yaml
---
type: source
origin: "{포스트 폴더명}"
kind: "{글의 종류}" # 선택 필드. 모호하면 비워둠
---
```

- `origin`: 원본 포스트 폴더명. `raw/{origin}/index.mdx`로 원본에 접근한다.
- `kind`: 글의 성격. 아래 중 하나. 경계가 모호하면 생략 가능.
  - `지식` — 외부 소스(글, 강연, 유튜브, 책 등)의 지식을 정리한 것. 내 생각이 덧붙을 수 있음
  - `회고` — 일정 기간의 경험을 돌아보는 글
  - `탐구` — 기술/개념을 조사하고 정리한 글
  - `에세이` — 본인의 고유한 생각과 주장
  - `가이드` — 실용적인 팁, 방법론 정리

### 본문 구조

```markdown
## 요약

포스트 핵심 내용 3-5문장 요약.

## 핵심 생각

이 포스트에서 드러나는 저자의 관점, 주장, 판단.
팩트 나열이 아니라 "이 글을 쓴 사람은 이렇게 생각한다"가 드러나야 한다.
`지식` kind라면 원저자의 주장과 나의 생각을 구분하여 서술한다.
- 후기나 본인 의견이 포함된 경우: 원저자의 주장을 정리한 뒤, 본인의 관점을 명시적으로 구분하여 서술한다.
- 후기 없이 정리만 한 경우: 원저자의 주장만 정리하고, "저자 본인의 의견은 명시되지 않음"이라고 밝힌다. 이 글을 선택하여 정리했다는 것은 관심 영역의 근거로 활용할 수 있지만, 본인이 동의한다고 귀속해서는 안 된다.

## 접점

다른 포스트, topic과의 관계. 생각이 어떻게 이어지거나 변화했는지.

- [다른 포스트](./other-post.md) — 관계 설명
- [주제 페이지](../topics/topic-name.md) — 이 포스트가 해당 주제에 기여하는 바

## 성장 기록

이 포스트에서 드러나는 성장의 지점. 새로 알게 된 것, 관점의 변화, 시도한 것.
모든 포스트에 해당하지는 않으므로, 드러나는 것이 없으면 이 섹션을 생략한다.
```

## Topic 페이지 (`topics/{topic-name}.md`)

여러 source를 하나의 주제로 묶고, "이 주제에 대해 나는 어떤 사람인가"에 답하는 문서. topic은 미리 정하지 않고, ingest 과정에서 포스트 내용을 보고 생성하거나 기존 topic에 통합한다.

### frontmatter

```yaml
---
type: topic
---
```

### 본문 구조

```markdown
## 개요

이 주제에 대한 종합 설명. 여러 포스트를 관통하는 관점.

## 관련 포스트

- [폴더명](../sources/folder-name.md) — 이 주제를 어떻게 다루는지

## 변화와 흐름

이 주제에 대한 생각이 시간순으로 어떻게 변해왔는지.
시리즈가 있다면 여기서 순서와 흐름을 설명한다.

## 인사이트

포스트들을 종합했을 때 발견되는 패턴, 경향, 탐구할 만한 질문.
각 인사이트에는 근거가 되는 포스트를 명시한다.

- 인사이트 내용
  근거: [포스트1](../sources/post1.md), [포스트2](../sources/post2.md)

## 관련 주제

- [다른 주제](./other-topic.md) — 관계 설명
```

## index.md

전체 wiki 페이지의 카탈로그. 카테고리별로 정리하며, 각 항목은 링크 + 한줄 설명.

```markdown
# Wiki Index

## Sources (N개)
- [폴더명](sources/folder-name.md) — 한줄 요약

## Topics (N개)
- [주제명](topics/topic-name.md) — 한줄 설명
```

## log.md

시간순 append-only 기록. 각 항목의 prefix는 파싱 가능한 형식을 따른다.

```markdown
# Wiki Log

## [YYYY-MM-DD] ingest | folder-name @ commit-hash
어떤 페이지들이 생성/수정되었는지. commit-hash는 ingest 시점의 HEAD를 기록한다.

## [YYYY-MM-DD] query | 질문 요약
어떤 페이지를 참조했는지, 새 페이지를 만들었는지.

## [YYYY-MM-DD] lint
발견된 문제와 수정 내역.
```

## 크로스 레퍼런스 규칙

- 표준 markdown 상대 경로 링크: `[텍스트](../topics/topic.md)`
- 새 source를 추가하면 관련 topic 페이지도 갱신
- topic에서 source를 참조할 때는 그 포스트가 해당 주제를 어떻게 다루는지 한 줄 설명
- 양방향 링크 유지: A→B 링크가 있으면 B→A 링크도 있어야 함
