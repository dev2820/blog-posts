# wiki-lint

wiki의 구조적 건강 상태를 점검하고 문제를 수정한다.

## 사전 준비

1. `wiki/WIKI_SCHEMA.md`를 읽어 wiki 규칙을 파악한다.
2. `wiki/index.md`를 읽어 등록된 페이지 목록을 파악한다.

## 점검 항목

아래 항목을 순서대로 점검한다. 각 항목의 결과를 사용자에게 보고하고, 수정이 필요한 부분은 사용자 확인 후 수정한다.

### 1. source 누락 점검

`raw/` 하위의 모든 포스트 폴더를 조회하고, 대응하는 `wiki/sources/{folder-name}.md`가 존재하는지 확인한다.

- source가 없는 포스트 → ingest가 필요하다고 보고

### 2. 깨진 링크 점검

모든 wiki 페이지(sources/, topics/)의 markdown 링크를 파싱하여, 링크 대상 파일이 실제로 존재하는지 확인한다.

- 존재하지 않는 파일을 가리키는 링크 → 보고 후 수정 또는 제거

### 3. 양방향 링크 점검

source A가 source B를 참조하면, B도 A를 참조해야 한다. topic↔source도 마찬가지.

- 한쪽만 참조하는 경우 → 역방향 링크 추가

### 4. index.md 일관성 점검

`wiki/index.md`에 등록된 페이지 목록과 실제 `wiki/sources/`, `wiki/topics/` 디렉토리의 파일들을 대조한다.

- 실제 파일은 있는데 index에 없는 경우 → index에 추가
- index에는 있는데 파일이 없는 경우 → index에서 제거
- 개수(N개) 불일치 → 갱신

### 5. topic 건강 점검

각 topic 페이지를 읽고 다음을 확인한다:

- "관련 포스트" 섹션에 나열된 source가 실제로 존재하는지
- 관련 포스트가 1개뿐인 topic → 다른 포스트와 병합 가능한지 검토
- "인사이트" 섹션에 근거가 명시되어 있는지

### 6. 새 topic 제안

기존 source들을 훑어보고, 아직 topic으로 묶이지 않았지만 공통 주제가 있는 source 그룹을 발견하면 새 topic 생성을 제안한다.

### 7. frontmatter 점검

모든 wiki 페이지의 frontmatter가 WIKI_SCHEMA.md에 정의된 형식을 따르는지 확인한다.

- `type` 필드 누락 또는 잘못된 값
- source 페이지의 `origin` 필드가 실제 raw 폴더와 일치하는지

## 결과 보고

점검 결과를 다음 형식으로 사용자에게 보고한다:

```
## wiki-lint 결과

### 문제 발견 (N건)
- [ ] {문제 설명} — {대상 파일}

### 제안 (N건)
- {제안 내용}

### 요약
- source: {N}개 중 {M}개 정상
- topics: {N}개 중 {M}개 정상
- 깨진 링크: {N}건
- 누락된 양방향 링크: {N}건
```

## 수정

사용자가 수정을 승인하면 해당 파일들을 수정한다. 수정 후 `wiki/log.md`에 기록한다:

```markdown
## [YYYY-MM-DD] lint

{수정 내용 요약}. 깨진 링크 N건 수정, 양방향 링크 N건 추가, index.md 갱신.
```

## 주의사항

- `raw/`의 어떤 파일도 수정하지 않는다.
- 수정은 항상 사용자 확인 후 진행한다.
- 대량 수정이 필요하면 수정 대상 목록을 먼저 보여주고 일괄 승인을 받는다.
